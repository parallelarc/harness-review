大家好，今天这场分享的主题是Claude Code Harness。

我们在过去一年多里经历了一个明显的范式迁移：最早大家做 Prompt Engineering，本质上是精心设计单次交互；后来演变成 Context Engineering，关注的是如何在有限的上下文窗口里组织好信息；到现在，我们谈的是Harness——一套完整的工程化体系，把 AI Agent 当成一个确定性生产系统的一部分来管理。

今天的内容会覆盖几个核心模块：Agent 的本质循环和五种经典模式、Claude Code 的权限系统如何避免越界、上下文工程的构成要素、四级记忆系统、以及 Skill 和 Subagent 的设计哲学。

大约 30 分钟，节奏会比较紧凑。有问题可以随时打断，也可以留到最后 Q&A。

01 / 背景

为什么需要 Harness？

范式的三次跃迁

Prompt Engineering

精心设计单次交互的提示词。模型能力有限时，人在替模型做决策。

Context Engineering

在有限的上下文窗口里组织信息。模型变强了，瓶颈转移到信息供给。

Harness

完整的工程化体系——权限、记忆、技能、子 Agent。默认模型已经够强，人类逐渐成为效率瓶颈。

核心理念

Harness 代表一种观念转变：默认模型能力已经很强，而人类逐渐成为系统效率的瓶颈。

先聊聊背景。为什么要引入 Harness 这个概念？

我们把时间线拉长来看。第一阶段是 Prompt Engineering，那时候模型能力有限，你需要非常精心地设计每一句提示词，本质上是在替模型做决策。

第二阶段是 Context Engineering。模型变强了，但它能看到什么、看不到什么，成了关键问题。你需要在有限的上下文窗口里高效地组织信息。

到了第三阶段就是 Harness。模型能力已经足够强，我们不再需要花大量精力去"教"模型，而是需要一套工程体系来管理权限、记忆、技能和子 Agent。核心观念变了——人类逐渐成为系统效率的瓶颈。

这不是说人不重要了，而是说人的角色从"操作者"变成了"系统设计师"。

那 Agent 到底是什么？抛开所有复杂的框架和术语，Agent 的本质就是一个 while(true) 循环：感知、决策、行动、反馈，然后回到感知。

这个循环听起来简单，但关键在于你怎么组合它。业界总结了五种经典的 Agent 模式，大多数 Agent 系统都是这五种模式的组合。

左边这张图展示了 Agent Loop 的基本结构，右边这张展示了模式如何组合。大家先有个整体印象，下一页我会逐个拆解这五种模式。

记住一个核心观点：好的 Agent 设计不是发明新模式，而是正确地组合已有模式。就像搭积木一样，组件是有限的，但组合方式是无限的。

02 / Agent Loop

五种经典模式

大多数 Agent 系统是这些模式的组合

① 提示链 Prompt Chaining

任务拆成顺序步骤，每步处理上一步输出。中间可加代码检查点。适合生成后翻译、先大纲再正文。

② 路由 Routing

对输入分类，定向到专用处理流程。简单问题走轻量模型，复杂问题走强模型。

③ 并行 Parallelization

分段法：任务拆成独立子任务并发；投票法：同一任务跑多次取共识。适合高风险决策。

④ 编排器-工作者

中央 LLM 动态分解任务，委派给工作者，综合结果。Claude Code 的 spawn 和子 Agent 就是这个模式。

⑤ 评估器-优化器

生成器产出，评估器给反馈，循环直到达标。适合翻译、创意写作等质量标准难以精确定义的任务。

Claude Code 主要使用模式④（Orchestrator-Workers）+ 模式③（Parallelization for Subagents）

这五种模式大家不需要都背下来，但需要理解它们的适用场景。

提示链最简单，适合线性的、步骤明确的任务。比如先让模型写大纲，确认后再写正文。路由本质是一个 if-else——根据输入类型分发到不同处理逻辑。

并行有两种用法：一种是把大任务拆成小块同时跑，另一种是同一件事跑多次投票取共识，这在高风险决策场景很有用。

编排器-工作者是最重要的一个模式，也是 Claude Code 的核心。主 Agent 作为编排器动态分解任务，委派给子 Agent，最后综合结果。

评估器-优化器适合质量标准难以精确描述的场景，比如翻译质量——让另一个模型来评估，不达标就重新生成。

实际系统中，这些模式往往是嵌套组合的。Claude Code 就同时使用了编排器-工作者和并行模式。

03 / 权限系统

Claude Code 可见内容与权限边界

完善的机制避免越界

七种权限模式

type PermissionMode = | 'default' // 敏感操作始终询问 | 'acceptEdits' // 自动批准文件编辑 | 'bypassPermissions' // 自动批准一切 ⚠️ | 'dontAsk' // 自动拒绝需要询问的 | 'plan' // 只读 + 计划文件 | 'auto' // AI 分类器自动审批 | 'bubble'; // 冒泡到父 Agent

Claude Code 能看到什么？

项目文件 — 目录及子目录中的文件

终端 — 你能运行的命令它都能跑

Git 状态 — 分支、未提交更改、提交历史

CLAUDE.md — 人类引导行为的项目契约

Auto Memory — 自动保存的学习成果

聊完 Agent 模式，我们来看看 Claude Code 的权限系统。这一块经常被忽略，但它其实是 Harness 能安全运行的基础。

左边是七种权限模式，日常主要用四种：default 是最安全的，敏感操作都会询问你；acceptEdits 自动批准文件编辑但其他操作还是询问；bypassPermissions 就是大开后门，一般只在 CI 或完全信任的场景用；plan 模式很有意思——它让 Agent 只能读文件和写计划，不能改任何代码。

右边是 Claude Code 能看到的所有内容。注意，它能看到的东西和你一样——项目文件、终端命令、Git 状态。还有一个很重要的 CLAUDE.md，它是你给 Agent 的项目契约。

所以权限设计的核心思路是：默认限制，按需放开。Bubble 模式是给子 Agent 用的——子 Agent 遇到越权操作时，不是直接拒绝，而是冒泡回主 Agent 处理。

04 / Context Engineering

Agent 看不到的内容等于不存在

上下文的构成

始终常驻
CLAUDE.md

项目契约 / 构建命令 / 禁止事项

按路径加载
Rules

语言 / 目录 / 文件类型特定规则

按需加载
Skills

工作流 / 领域知识

隔离加载
Subagents

大量探索 / 并行研究

不进上下文
Hooks

确定性脚本 / 审计 / 阻断

这句话我希望大家记住："Agent 看不到的内容等于不存在。"这就是 Context Engineering 的核心。

左边列出了上下文的五种加载方式。最上面是始终常驻的 CLAUDE.md，每次对话都会加载，相当于项目的基本法。然后是按路径加载的 Rules，比如你打开一个 .py 文件，就自动加载 Python 相关的规则。

Skills 是按需加载的——只有在需要时才注入上下文，不浪费宝贵的 token。Subagents 是隔离加载的——每个子 Agent 有独立的上下文窗口，互不污染。

最后是 Hooks，这个很特别——它们根本不进上下文窗口。Hooks 是确定性脚本，在工具调用前后执行，Agent 完全感知不到。这就像一个隐形的护栏。

右边这张图展示了这些元素如何协同工作。核心原则是渐进式披露——不是把所有信息一股脑塞进去，而是按需、按路径、按场景加载。

05 / 记忆系统

通用四级记忆系统

从工作记忆到语义记忆

L1 上下文窗口
工作记忆

当前任务所需的最小信息。Token 有限，需主动管理。超过 50% 后性能指数级下降。

L2 Skills
程序性记忆

怎么做某件事——操作流程、领域规范。按需加载，不默认常驻。

L3 会话历史
情景记忆

JSONL 磁盘持久化。发生了什么——支持跨会话检索。

L4 MEMORY.md
语义记忆

Agent 主动写入认为重要的事实。每次启动时注入系统提示，前 200 行或 25KB。

OpenClaw 混合检索：日志追加写 + 精选事实 + 70% 向量相似度 + 30% 关键词权重

记忆系统是 Harness 的另一个核心模块。我把它类比成人类的四级记忆。

L1 是工作记忆，就是上下文窗口。这个窗口很小，而且有一个很重要的发现：超过 50% 之后性能会指数级下降。所以你为什么要频繁 /compact、为什么要管理会话——不是因为技术限制，而是因为性能会变差。

L2 是程序性记忆，就是 Skills。类比你会骑自行车这个能力——你不会一直想着怎么骑车，但需要的时候自然就用上了。Skills 也是这样，按需加载。

L3 是情景记忆，就是会话历史。以 JSONL 格式持久化到磁盘，记录了"发生了什么"。支持跨会话检索。

L4 是语义记忆，就是 MEMORY.md。Agent 会主动把认为重要的事实写进去，每次启动时自动加载。这是最持久的记忆层。

OpenClaw 还引入了混合检索机制：70% 向量相似度加 30% 关键词权重，兼顾语义理解和精确匹配。

06 / Skills

记忆记住"是什么"，技能记住"怎么做"

Tools 是原子操作，Skills 是工作流

Tools vs Skills

Tools = 原子操作（读文件、运行命令）

Skills = 工作流（"帮我做 Code Review"、"帮我部署到 staging"）

Skill 描述三要素

  • Use when — 什么时候用
  • Don't use when — 什么时候不用
  • 反例是必要的（准确率从 53% 提升到 85%）

频次分层

高频 → 常驻系统提示 · 低频 → 按需加载 · 极低频 → 直接用文档替代

接下来是 Skills。前面说了记忆是记住"是什么"，那 Skills 就是记住"怎么做"。

一个关键的区分：Tools 是原子操作，比如读一个文件、运行一条命令；Skills 是工作流，比如"帮我做一次完整的 Code Review"或者"帮我部署到 staging 环境"——这背后可能涉及十几个 Tool 调用。

Skill 的描述有一个很重要的设计原则：必须包含反例。有一项实验表明，没有反例时准确率只有 53%，加上反例后跳到 85%，而且响应时间还降了 18%。所以"什么时候不用"和"什么时候用"一样重要。

另外一个实用建议是按频次分层：高频 Skill 放在系统提示里常驻，低频的按需加载，极低频的干脆就用文档替代，不要做成 Skill。因为每个常驻 Skill 都在消耗你的上下文窗口。

07 / Subagents

Orchestrator-Workers 模式

上下文隔离，独立并行

主 Agent

Orchestrator

↓

Subagent A

读大文件

Subagent B

Web Search

Subagent C

代码探索

↓ 摘要返回（非完整上下文）

优势一：避免权限泄露

子 Agent 从空白消息列表开始，不继承主 Agent 的敏感上下文。

优势二：避免上下文污染

子 Agent 的完整输出不回灌主 Agent，只返回精炼摘要。

Subagent 是 Claude Code 中 Orchestrator-Workers 模式的直接体现。

主 Agent 作为编排器统筹全局，需要做大文件读取、Web 搜索、代码探索等重操作时，就委派给子 Agent。子 Agent 完成后只返回摘要，不是把整个上下文倒回来。

这个设计有两个核心优势。第一，避免权限泄露：每个子 Agent 从空白消息列表开始，它看不到主 Agent 之前的对话，所以即使子 Agent 行为异常，也不会泄露敏感信息。

第二，避免上下文污染：想象一下，如果一个子 Agent 读了一个 2000 行的文件，直接把内容灌回主 Agent 的上下文窗口，那主 Agent 基本就废了。只返回摘要，既保留了关键信息，又保护了主 Agent 的上下文空间。

所以上下文隔离不只是安全措施，更是性能措施。这也是为什么 Harness 要把 Subagent 作为一个一等公民来管理。

08 / 最佳实践

大多数最佳实践都基于一个限制

上下文窗口很快就会填满

  1. 会话管理，防止 context rot

    rewind / 及时 /compact / HANDOFF.md + 新开对话 / 方向不对直接新开比纠正更高效

  2. Explore first, then plan, then code

    标准工作流：spec → plan → implement → test，提供明确的验收标准

  3. 确定性的逻辑交给确定性的工程

    能用代码解决的不用 LLM，能用 Hook 拦截的不用提示词

  4. 渐进式披露

    把 Agent 当人类来看——提供足够明确的信息它才能工作，明确能力边界

  5. CLAUDE.md 太长不如不写

    Unnecessary requirements from context files make tasks harder

  6. Entropy Management

    时常整理代码库避免熵增，干净的代码库 = 高效的 Agent

最佳实践这一页是实打实的经验总结，每一条背后都有血泪教训。

所有实践的共同前提是：上下文窗口很快就会填满，超过 50% 后性能指数级下降。所以：

第一条，会话管理。context rot 是真实存在的问题——对话越长，Agent 越容易"忘记"之前的约定。遇到方向不对的情况，直接新开对话比纠正更高效。这违反直觉，但非常实用。

第三条很关键——确定性的逻辑交给确定性的工程。能用代码解决的问题不要让 LLM 来猜，能用 Hook 拦截的不要用提示词。LLM 擅长处理不确定性，但确定性的事情它反而不如脚本可靠。

第五条是我最喜欢的——CLAUDE.md 太长不如不写。有一篇论文量化证明了这一点："不必要的上下文文件要求会让任务变难。"所以 CLAUDE.md 应该只写最精简的必要信息。

最后总结一下今天最核心的几个 Takeaway。

Agent 的本质就是一个循环——感知、决策、行动、反馈。所有的复杂系统都是五种基本模式的组合。

上下文工程是一切的基础。Agent 看不到的东西等于不存在，所以信息组织的方式直接决定了系统效率。

确定性归确定性——能用代码和 Hook 解决的问题，不要让 LLM 来猜。

保持精简——CLAUDE.md 太长不如不写，unnecessary requirements make tasks harder。

最后，人类的角色在变化——从操作者变成系统设计师。Harness 不是取代人，而是让人能更高效地设计和管控 AI 系统。
